The pipeline

Six steps from idea to live database

No SQL expertise required. Describe your product in plain English — the skill handles entity design, normalization, provisioning, and code generation end-to-end.

01
🧠

Entity Extraction

Claude reads your spec and identifies every actor, resource, transaction, relationship, and event before writing any SQL.

02
📐

Schema Design

Full PostgreSQL DDL in 3NF — UUID PKs, proper FK policies, NUMERIC for money, TIMESTAMPTZ everywhere, and indexes on every FK.

03
🗄️

Creates the target database automatically from the business name. Every DDL statement uses IF NOT EXISTS so re-runs never crash.

Requirements & setup

Connect via MCP in three steps

Required

✓ Claude Desktop — with MCP support enabled (Settings → Developer)
✓ Datafy MCP Server — @teckedd-code2save/datafy registered in claude_desktop_config.json
✓ PostgreSQL instance — local or remote, defined as a source in your dbhub.toml
✓ SKILL.md — placed in your Claude skills directory so the skill auto-triggers

MCP tools Datafy exposes

✓ execute_sql_<id> — run DDL or DML against any configured postgres source
✓ search_objects_<id> — inspect schemas, tables, columns, indexes
✓ generate_code — converts SQL to TypeScript or C# (Prisma, Dapper, EF Core)
○ redis_command_<id> — optional session/cache store
○ elasticsearch_search_<id> — optional logs and analytics store
Step 1 — Install Datafy
$ npm install -g @teckedd-code2save/datafy
Step 2 — Add to claude_desktop_config.json
{
  "mcpServers": {
    "datafy": {
      "command": "npx",
      "args": [
        "@teckedd-code2save/datafy@latest",
        "--config",
        "/path/to/your/dbhub.toml",
        "--transport",
        "stdio"
      ]
    }
  }
}
Step 3 — Create your dbhub.toml
# dbhub.toml — define all your data sources
# PostgreSQL (required)
[[sources]]
id = "pet_market" # → execute_sql_pet_market, search_objects_pet_market
type = "postgres"
host = "localhost"
port = 5432
database = "petty"
user = "postgres"
password = "postgres"
lazy = true
# Redis — session / cache (optional)
[[sources]]
id = "session_storage" # → redis_command_session_storage
type = "redis"
host = "localhost"
port = 6379
database = "0"
lazy = true
description = "Redis for session storage"
# Elasticsearch — logs & analytics (optional)
[[sources]]
id = "logs_and_analytics" # → elasticsearch_search_logs_and_analytics
type = "elasticsearch"
host = "localhost"
port = 9200
lazy = true
description = "Elasticsearch for logs and analytics"
💡 Each [[sources]] id becomes the MCP tool suffix — so "pet_market" automatically exposes execute_sql_pet_market and search_objects_pet_market to Claude.

Skills · Database · AI

From “build me a ride-sharing app” to a live PostgreSQL database

March 2026 · 8 min read

Most database tutorials start at the schema. But when a non-technical founder describes their product, they don’t think in tables — they think in actors, actions, and outcomes. This skill bridges that gap.

We walk through the full pipeline: how entity extraction prevents schema drift, why every monetary value must be NUMERIC(12,2) and never FLOAT, what makes seed data feel real versus synthetic, and the two bugs we fixed to make provisioning actually work — database creation, and the nuclear importance of IF NOT EXISTS on every single DDL statement.

If you’ve ever watched a beautifully designed schema fail on a second run because someone forgot CREATE TABLE IF NOT EXISTS, this one’s for you.
