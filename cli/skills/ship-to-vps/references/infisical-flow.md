# Infisical Lifecycle Across forge + ship-to-vps

Infisical is the single source of truth for every project secret. Everything else (GitHub Variables, GitHub Secrets, VPS `.env`) is a **projection** of Infisical state. When secrets change, projections must re-sync.

The user runs a self-hosted Infisical at **https://<your-infisical-domain>**. See `~/.claude/rules/secrets.md` for the canonical wrapper conventions (`sec`, `~/.infisical/projects/<slug>.env`).

---

## The 6 phases

### Phase 0 — Project bootstrap (forge Step 0)
**Owner:** forge

For every new project, forge uses the bootstrap identity to:

1. `POST /v2/workspace` → create Infisical project (slug = repo name) → capture `projectId`
2. `POST /v3/secrets/raw/<KEY>` → seed initial secrets, copying shared values from the `shared` project as placeholders
3. `POST /v1/identities` → create org identity `<slug>-agent`
4. `POST /v2/workspace/<projectId>/identity-memberships/<identityId>` → attach as member
5. `POST /v1/auth/universal-auth/identities/<identityId>` → configure Universal Auth (TTL 30d, IP 0.0.0.0/0)
6. `POST /v1/auth/universal-auth/identities/<identityId>/client-secrets` → mint client secret
7. Add the human admin user to the project
8. Write `~/.infisical/projects/<slug>.env` (chmod 600)
9. Write `.infisical.json` in the repo root

After this, `sec` in the repo Just Works.

### Phase 1 — Local dev (always)
**Owner:** user (via `sec`)

```bash
sec -- npm run dev          # auto-detects project from .infisical.json
sec staging -- npm run build
```

No `.env` file is ever written to disk locally. The wrapper logs in via Universal Auth, mints a short-lived access token, runs `infisical run` which injects secrets as process env vars only.

### Phase 2 — CI build-time
**Owner:** ship-to-vps (via `deploy.yml`)

Only **build-time public values** (`NEXT_PUBLIC_*`) reach the CI runner. They live in **GitHub Variables**, mirrored from Infisical at ship-to-vps setup time. They pass into the Docker build as `--build-arg` and get inlined into the client bundle.

Setup-time mirror command:
```bash
for KEY in $(sec <slug> prod -- sh -c 'env | grep ^NEXT_PUBLIC_ | cut -d= -f1'); do
  VAL=$(sec <slug> prod -- sh -c "printenv $KEY")
  gh variable set "$KEY" --repo <owner>/<repo> --body "$VAL"
done
```

Drift: if `NEXT_PUBLIC_*` changes in Infisical, re-run the mirror command (or let `infisical-sync.yml` handle it). The next deploy bakes the new value.

### Phase 3 — CI deploy-time
**Owner:** ship-to-vps (via `deploy.yml`)

The deploy job needs three secrets from **GitHub Secrets**:
- `VPS_SSH_KEY` — private SSH key for the deploy user
- `VPS_HOST` — VPS IP or hostname
- `VPS_USER` — usually `root`

GHCR auth uses ephemeral `${{ secrets.GITHUB_TOKEN }}` with `packages: read` permission — works as long as the GHCR package is linked to the repo (auto-linked when the image has the OCI source label).

### Phase 4 — VPS runtime
**Owner:** Docker Compose on VPS

The web container is started with `--env-file /opt/<slug>/.env`. That file is the **only** place secrets exist as a file on the VPS. It is `chmod 600`, owned by root, generated by Phase 5.

`docker-compose.yml` on the VPS references it via:
```yaml
env_file:
  - .env
```

### Phase 5 — Drift sync (scheduled)
**Owner:** ship-to-vps (via `infisical-sync.yml`)

Runs **hourly** and on `workflow_dispatch`. The workflow:

1. Authenticates to Infisical with the project's Universal Auth identity (creds in **GitHub Secrets** as `INFISICAL_CLIENT_ID` + `INFISICAL_CLIENT_SECRET`)
2. Fetches all `prod`-env secrets → renders a candidate `.env`
3. SSHes to VPS, computes `sha256` of current `/opt/<slug>/.env` and the candidate
4. If different: writes new file atomically, `docker compose up -d` to restart with fresh env
5. If same: no-op, exit clean
6. Also re-mirrors `NEXT_PUBLIC_*` to GitHub Variables (so the next build picks them up)

Audit trail: every sync run is in the Actions log with the diff hash.

## Anti-patterns

- ❌ Pasting a real secret into a `.env` file
- ❌ Adding a real secret to GitHub Secrets (use Infisical, then sync)
- ❌ Hardcoding `process.env.X = "..."` defaults that mask missing secrets
- ❌ Using the bootstrap identity for app runtime (it's admin-level, provisioning only)
- ❌ Storing the Infisical client secret in the VPS `.env` (it's needed *before* the .env exists; it's a GitHub Secret only)
